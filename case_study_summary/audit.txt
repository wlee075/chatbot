Q1. Do we already have any complexity scoring logic per response?
================================================================================
Status: NO

No function, field, or heuristic anywhere in the codebase computes a
"complexity score" for an individual chatbot response.

The closest candidate is `_answer_confidence_score` in
  graph/nodes.py:2033
but this scores the USER'S ANSWER quality (word count, specificity), NOT the
response risk or complexity. It feeds into `section_draft_meta` to decide
whether to START drafting — a drafting-policy gate, not a human-oversight gate.

There is also `missing_required_fields_count` and `draft_readiness_band`
(state.py:69-71) which classify section completeness into bands like "Ready",
"Near Ready", "Blocked". These drive `next_action` routing (nodes.py:2998),
but they measure SECTION readiness, not per-response risk.

Wiring to oversight: NONE. These scores never gate user-facing approval.

================================================================================
Q2. Do we already have any risk scoring logic per response?
================================================================================
Status: NO

No field or function named "risk_score", "risk_level", or equivalent exists.

`impacted_section_scores` (state.py:218, nodes.py:2389) assigns confidence
floats to cross-section impact (1.0 = rule-based, 0.7 = LLM-fallback,
0.4 = side-fact). These are used in draft_node (nodes.py:2594-2598) to skip
low-confidence side-writes below `_SIDE_WRITE_MIN_IMPACT_SCORE = 0.5`.

This is an internal drafting quality gate, NOT a user-facing risk assessment.
It never triggers approval UI or blocks state transitions pending human review.

================================================================================
Q3. Do we classify responses into multiple autonomy modes today?
================================================================================
Status: NO

No "mode_tag", "auto_proceed", "confirm_before_send", or "user_must_lead"
field or enum exists anywhere in the codebase.

The system has structural routing modes:
  - next_action: "START_DRAFT" | "ASK_ONE_MORE" | "ASK_MULTIPLE" |
    "UPDATE_DRAFT" | "ADVANCE_SECTION" | "WAITING_CONFIRMATION"
    (state.py:67, nodes.py:2998-3009)

These describe WORKFLOW steps, not autonomy/oversight levels. The system
always operates at a single, fixed autonomy level: the chatbot acts and the
user passively responds. There is no mechanism to switch between
"auto-proceed" vs. "confirm-before-send" vs. "user-must-lead" per response.

================================================================================
Q4. Do we already block or require confirmation before risky state transitions?
================================================================================
Status: PARTIAL — but NOT risk-based

The echo/confirmation gate exists:
  - `interpret_and_echo_node` (nodes.py:1484) restates the user's answer.
  - `answer_confirmation_status` cycles through "" → "PENDING" → "CONFIRMED"
    or "CORRECTED" (state.py:173, nodes.py:1186, 1348, 1797, 1977, 1997).
  - `route_after_confirmation` (routing.py:109) routes CONFIRMED → draft,
    CORRECTED → await_answer.

CRITICAL DISTINCTION: This gate confirms FACT ACCURACY ("did I hear you
correctly?"), NOT risk governance ("this action is high-risk, are you sure?").
Every single answer goes through the exact same echo gate regardless of risk.
There is no conditional path where high-risk responses get MORE scrutiny.

The bounds-check gate (nodes.py:1501-1511, utils/validator.py) catches
impossible numeric inputs (e.g. team_size = 99999) via rule-based
plausibility checks. This IS a risk gate, but only for a narrow numeric
case. It does NOT generalize to arbitrary response risk.

Q5. Do we already have approve/edit/reject UI controls for assistant outputs?
================================================================================
Status: NO

app.py contains NO approve, reject, or edit buttons for assistant-generated
content. The only st.button instances are:
  - Line 552: unclear context (checked — it is for sending chat input)
  - Line 671: "↩ New" — resets session
  - Line 752: section preview toggle
  - Line 776: "Close" preview
  - Line 994: "✕" — clear active reference

The TAG_MESSAGE_AS_TRUTH and CORRECT_MESSAGE events are user-initiated
actions (clicking on a past message to promote/correct it). They are NOT
approval gates on assistant output. The assistant's responses render
immediately with no interception point.

================================================================================
Q6. Do we log user acceptance/rejection of assistant actions?
================================================================================
Status: PARTIAL — correction stats only, no acceptance/rejection logging

`correction_stats` (state.py:186) tracks success/failure counts for
CORRECT_MESSAGE events (nodes.py:1353-1356). This logs whether a correction
linkage succeeded, NOT whether a user approved or rejected an assistant
recommendation.

`event_history` (state.py:185) logs TAG_MESSAGE_AS_TRUTH and CORRECT_MESSAGE
events with target_message_id, content, section, concept_key. This is a
provenance log, not an acceptance/rejection log.

The JSONL logger (utils/logger.py) logs routing_decision, node_start,
node_end, suppression_observability, etc. None of these event_types capture
user-level approval/rejection decisions on assistant output.

  extracted terms, not whether the overall response is risky to commit)
- The echo gate is NOT a "confirm-before-send" mode — it fires
  unconditionally on every answer, providing zero risk differentiation

================================================================================
BOTTOM LINE
================================================================================
This is NOT already in the system. We are flattering ourselves.

The codebase has strong foundational primitives — contradiction detection,
confidence scoring, provenance tracking, echo gating, structured logging —
but NONE of them are wired into a unified oversight framework. They each
serve narrow, independent purposes (drafting policy, duplicate suppression,
provenance display). The gap between "we have useful signals" and "we have
adaptive human oversight" is the entire classifier + conditional gate + UI
+ logging pipeline. That pipeline does not exist today.
